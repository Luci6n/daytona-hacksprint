# DaddyFix backend API integration guide

This document is the client contract for Kenji's `VisionService.swift` and
`VoiceManager.swift`. The canonical response models are defined in
`backend/models.py` and mirrored in
`DaddyFix/DaddyFix/Models/AnalysisResult.swift`.

## Base URL and interactive documentation

Local development base URL:

```text
http://127.0.0.1:8000
```

Replace it with the public Daytona/proxy URL during device integration. The
API currently has no client authentication. Sponsor credentials remain on the
server and must never be sent by the iOS app.

- Swagger UI: `GET /docs`
- OpenAPI JSON: `GET /openapi.json`
- API version: `0.3.0`

## End-to-end iOS flow

1. Swift captures a JPEG frame and optional device hint.
2. Apple Speech converts microphone audio into a transcript on the iPhone.
3. For one-shot mode, `VisionService.swift` sends the JPEG and transcript to
   `POST /analyze`.
4. For live mode, Swift keeps one `WS /live/{sessionId}` connection, samples
   camera JPEGs, and sends final utterances over that connection.
5. The backend returns `AnalysisResult`; ARKit consumes `arAnnotations`.
6. The backend returns a WAV generated by Qwen3-TTS on Nosana; Swift plays it.

Apple Speech recognition is an iOS responsibility. Do not upload raw microphone
audio to `/analyze`; send the recognized text in `symptom`.

Use this before the demo and when selecting a backend URL.

```json
{

  "service": "daddyfix-backend",
  "environment": "development",
  "demoMode": true,
  "providers": {
    "daytona": true,

    "oxylabs": true,
    "doubleword": true,
    "aiand": true

```

Provider values indicate configuration presence, not a live network probe.
No keys, passwords, or endpoint tokens are returned.

## `POST /analyze`

Analyzes a captured repair scene and returns the shared `AnalysisResult`.

### Request

`Content-Type: application/json`

```json
{
  "symptom": "No hot water",
  "deviceHint": "Rinnai tankless water heater",
  "imageBase64": "data:image/jpeg;base64,/9j/4AAQSk..."
}
```

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `symptom` | string or null | No | Apple Speech transcript or typed symptom. |
| `deviceHint` | string or null | No | A visible brand/model hint improves manual search. |
| `imageBase64` | string or null | Live mode: yes | A data URL is recommended; raw base64 is also accepted as JPEG. |

In `DEMO_MODE=true`, all fields are optional and the deterministic water-heater
fixture is returned without spending provider credits.

For iOS, resize the captured image before encoding. A JPEG around 1280 px on its
long edge is sufficient for the demo and keeps request latency manageable.

### Live provider paths

- Moonshot path: Oxylabs context plus the image is sent directly to Kimi K2.6.
- ai& path: Doubleword Qwen3-VL first creates a visual observation; ai& Kimi
  K2.7 Code then reasons over that observation and the Oxylabs context.
- When Doubleword is configured, the final repair guidance receives an
  additional safety audit.

The backend never silently replaces a failed live call with demo data.

## `WS /live/{sessionId}`

This is the stateful camera-and-conversation contract. It does not appear in
FastAPI OpenAPI because OpenAPI documents HTTP routes, not WebSockets.

Local URL:

```text
ws://127.0.0.1:8000/live/kitchen-demo
```

Use `wss://` with the public Daytona/proxy host. Each connection owns its newest
camera frame, device hint, and previous completed `AnalysisResult`. State is
currently in-memory and is lost when the socket or backend process restarts.

### Message sequence

After connecting, the server sends:

```json
{ "type": "ready", "sessionId": "kitchen-demo" }
```

Swift samples the camera and sends a JPEG data URL. This replaces the previous
passive frame; it is not appended permanently to conversation history.

```json
{
  "type": "frame",
  "imageBase64": "data:image/jpeg;base64,/9j/4AAQSk...",
  "deviceHint": "Rinnai tankless water heater"
}
```

The server acknowledges it:

```json
{ "type": "frameAccepted" }
```

When Apple Speech produces a final transcript, Swift sends:

```json
{ "type": "utterance", "text": "What should I check first?" }
```

The server responds in this order. `turnId` identifies every result belonging
to this generation.

```json
{ "type": "status", "state": "analyzing", "turnId": "4c71..." }
```

```json
{
  "type": "analysis",
  "turnId": "4c71...",
  "data": {
    "detectedItem": "Rinnai Tankless Water Heater",
    "confidence": 0.94,
    "issues": ["ELCB may have tripped"],
    "arAnnotations": [
      {
        "type": "highlight",
        "x": 0.42,
        "y": 0.58,
        "z": null,
        "width": 0.18,
        "height": 0.12,
        "label": "ELCB",
        "color": "#22C55E"
      }
    ],
    "repairSteps": [
      {
        "step": 1,
        "instruction": "Locate the ELCB highlighted on the unit.",
        "safetyNote": "Stop and call a licensed professional if anything looks damaged."
      }
    ],
    "buyableParts": []
  }
}
```

The real `data` value uses the complete `AnalysisResult` contract shown above.
This example keeps each array short only to make the event envelope readable.

```json
{ "type": "status", "state": "synthesizing", "turnId": "4c71..." }
```

```json
{
  "type": "audio",
  "turnId": "4c71...",
  "contentType": "audio/wav",
  "byteLength": 183204
}
```

The next WebSocket message is binary WAV data of exactly `byteLength` bytes.
The current service returns one complete WAV after synthesis; it does not yet
stream PCM/audio chunks.

### Frame cadence

- Keep ARKit/LiDAR rendering locally at its normal frame rate.
- Send at most one JPEG per second while the user is speaking.
- Send no idle frames for the simplest MVP, or one every 3–5 seconds.
- Always send a fresh keyframe when a turn starts, the user taps an AR object,
  or the transcript refers to “this”, “that”, or “here”.
- Resize to roughly 1280 px on the long edge before base64 encoding.

The evidence and comparisons behind this policy are in
[`live-camera-research.md`](live-camera-research.md).

### Interruption / barge-in

When Apple Speech detects credible user speech while DaddyFix audio is playing:

1. Swift immediately stops playback and clears queued audio locally.
2. Swift sends the current server-issued `turnId`:

```json
{ "type": "interrupt", "turnId": "4c71..." }
```

3. The server acknowledges:

```json
{ "type": "interrupted", "turnId": "4c71..." }
```

4. Swift sends the next **final** Apple Speech transcript as a new `utterance`.
5. Swift ignores any late event whose `turnId` is no longer current.

Stopping playback must happen before the network acknowledgement. Cancelling
the backend task prevents late delivery, but cannot forcibly stop an HTTP/GPU
call already executing in a worker thread. Configure iOS voice processing/echo
cancellation so the TTS voice does not trigger Apple Speech itself.

### WebSocket errors

```json
{
  "type": "error",
  "turnId": "4c71...",
  "code": "turn_failed",
  "detail": "Human-readable explanation"
}
```

`turnId` is absent for malformed client events. Current error codes are
`invalid_event`, `turn_failed`, and `internal_error`. Do not silently replace a
live error with local demo guidance.

## `POST /speech/synthesize`

Generates the configured DaddyFix voice through Qwen3-TTS VoiceDesign on Nosana.

### Request

```json
{
  "text": "Easy, sweetheart. Turn off the breaker first."
}
```

### Successful response

- Status: `200`
- Content-Type: `audio/wav`
- Body: binary WAV data

Swift should store the response in `Data` and play it using `AVAudioPlayer`.
Do not attempt to decode this endpoint as JSON.

The voice description, language, and model are server configuration. The iOS
client deliberately cannot override the persona prompt.

## Error contract

All JSON errors use this shape:

```json
{
  "detail": "Human-readable explanation"
}
```

| Status | Meaning | Suggested iOS behavior |
| --- | --- | --- |
| `422` | Invalid request or missing live-mode image | Ask for another capture or correct the request. |
| `502` | A provider failed, returned invalid data, or rejected unsafe guidance | Show the error and allow retry; do not invent guidance. |
| `503` | Required credentials or Nosana TTS URL are not configured | Disable the affected action and show backend unavailable. |

Use a client timeout of at least 60 seconds for `/analyze` and 120 seconds for
the first `/speech/synthesize` call because cold GPU model loading can be slow.
Warm the TTS service before the live demo.

## Swift client contract sketch

Kenji owns the actual `VisionService.swift`; this sketch documents the wire
format without modifying that file.

```swift
struct AnalyzeRequest: Encodable {
    let symptom: String?
    let deviceHint: String?
    let imageBase64: String?
}

let payload = AnalyzeRequest(
    symptom: transcript,
    deviceHint: "Rinnai tankless water heater",
    imageBase64: "data:image/jpeg;base64," + jpegData.base64EncodedString()
)

var request = URLRequest(url: baseURL.appendingPathComponent("analyze"))
request.httpMethod = "POST"
request.setValue("application/json", forHTTPHeaderField: "Content-Type")
request.timeoutInterval = 120
request.httpBody = try JSONEncoder().encode(payload)

let (data, response) = try await URLSession.shared.data(for: request)
let analysis = try JSONDecoder().decode(AnalysisResult.self, from: data)
```

Check the HTTP status before decoding `AnalysisResult`; non-2xx responses use
the error shape above.
